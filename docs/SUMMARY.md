# 架构设计总结

> Claude-Codex Multi-Agent 系统 — 核心设计要点

---

## 系统定位

一个**分层编排的多 Agent 协作开发流水线**，将用户需求自动拆解为功能模块、并行分析、整合为代码、自动审查、循环修复，最终交付完整可运行代码。

## 架构五层

```
用户交互层  →  编排调度层  →  专家 Agent 层  →  提示词工程层  →  代码实现层
   (User)       (Codex+SP)      (Experts)         (Prompt)         (Claude Code)
```

## 双阶段工作流

| 阶段 | 输入 | 处理 | 输出 |
|------|------|------|------|
| 阶段一：需求拆分 | 用户需求文本 | Codex 拆分 → Experts 分析 → Prompt Agent 整合 | 完整代码 |
| 阶段二：代码审查 | 完整代码 | Superpowers 分发 → Experts 审查 → Codex 评估 | 审查报告 / 修复指令 |

## 核心角色

| 角色 | 数量 | 职责 |
|------|------|------|
| Codex (主管) | 1 | 需求理解、任务编排、质量把控、冲突解决 |
| Superpowers 插件 | 1 | Agent 注册、消息路由、上下文注入、结果聚合 |
| Prompt Agent | 1 | 模块规格整合、依赖分析、Prompt 生成 |
| 专家 Agent | N (按模块) | 单模块需求分析、代码审查 |
| Claude Code | 1 | 代码生成、修复执行 |

## 四原则落地

| 原则 | 实现方式 |
|------|----------|
| 单一职责 | 每个 Expert Agent 只处理一个功能模块，input/output Schema 严格限定 |
| 最小权限 | Superpowers 上下文注入器只注入该模块的最小必要上下文 |
| 独立可测 | 每个 Agent 可独立启动，给定相同输入产出确定性输出 |
| 结果可验 | 所有输出必须通过 output_schema 校验，附带验收标准 |

## 关键设计决策

1. **Codex 作为唯一主管** — 避免多主管决策冲突
2. **Superpowers 中转所有通信** — 实现最小权限和隔离
3. **按功能模块分工** — 模块内聚更高，减少跨 Agent 依赖
4. **双阶段分离** — 需求分析和代码审查各自专注
5. **Prompt Agent 独立** — 解耦整合逻辑和决策逻辑
6. **Schema 强制校验** — 保证结果可验，防止错误传播
